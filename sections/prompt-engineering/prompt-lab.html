Prompt Engineering · Chapter 14

Prompt lab

The prompt lab is a hands-on exercise to write one prompt, test it against five labeled examples including one edge case, then revise it based on where it failed and re-test.

Calculating reading timeBuild project

What you're building, and why

In practice it comes down to writing an initial prompt, running it against a small labeled test set, identifying which cases failed and why, revising the prompt to address the specific failure, and re-testing to confirm the fix worked without breaking the cases that already passed.

It is like iterating on a recipe based on specific tasting feedback, not just guessing at improvements.

The brief

Write a prompt, test it against five labeled examples, revise it based on one specific failure, and re-test all five to confirm the fix didn't break anything else.

A worked reference

A lab might write a classification prompt, test it against five labeled examples, find it misclassifies a specific edge case, add an explicit instruction addressing that case, and re-test all five to confirm the fix worked without breaking the others.

Starting point in code

let prompt = "Classify this support ticket.";
let results = testSet.map(t => classify(prompt, t.input));
// found: misclassifies "urgent" vs "high priority" — revise and re-test
prompt += " Treat 'urgent' and 'high priority' as the same category.";

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Trade-offs to document

  • The core tension: Iterating based on a labeled test set takes more structured effort than tweaking a prompt based on a vague sense that the output looks better, but it's the only way to know a change actually improved things rather than just changed them

Definition of done

Watch out for

  • The lab is not complete once a revised prompt fixes the one failing case; you also need to re-test the cases that already passed, since a fix for one case can sometimes break another.
