Learning track

Evaluation

A production-focused path for mastering Evaluation through concepts, diagrams, hands-on labs, interview practice, and capstone work.

Track outcomes

By the end of this track, you should be able to reason from first principles, design the architecture, identify failure modes, build a working slice, and explain how the system behaves in production.

In plain words: Evaluation is like a driving test for an AI system. One good trip around the block is not enough; you need repeatable routes, edge cases, and scoring. Evals tell you whether a prompt, model, retrieval change, or guardrail made the product better or worse.

The running example

Throughout this track, imagine you own quality for 'Reply', a support assistant answering from your company's help center. Every chapter builds the measurement system around it — golden sets, gates, judges, dashboards — until 'is Reply getting better or worse?' has an answer that isn't an opinion.

Chapter 1 Eval strategy An evaluation strategy defines how you measure whether your AI system is producing good outputs — what data to test on, what metrics to compute, how often to run evals, and what threshold gates a release. Chapter 2 Golden datasets A golden dataset is a curated, versioned set of inputs with known-correct or preferred outputs — the fixed measuring stick that makes every model, prompt, and pipeline change comparable to the last one. Chapter 3 Unit evals Unit evals are small, fast, targeted checks that verify one specific behavior of an AI component — one prompt's format compliance, one extraction field, one refusal rule — the AI analogue of unit tests, run in seconds and wired into the developer loop. Chapter 4 Regression evals Regression evals compare a candidate change against the current baseline on the same fixed dataset — prompt edits, model upgrades, retrieval changes — to answer the only pre-ship question that matters: did anything that used to work stop working? Chapter 5 LLM-as-judge LLM-as-judge uses a language model (usually a stronger one) to evaluate the quality of another model's output, given a rubric. Chapter 6 Human review Human review is structured human judgment inside the eval loop — rubric-scored samples, side-by-side comparisons, and adjudication of disputed cases — the ground truth that calibrates every automated metric and catches what they can't see. Chapter 7 RAG evals RAG evals measure the pipeline in its separately-failing stages — did retrieval find the right evidence (recall, precision), and did generation use it honestly (faithfulness, citation accuracy) — because an end-to-end score can't tell you which half to fix. Chapter 8 Agent evals Agent evals measure multi-step behavior — did the agent reach the goal, via a sane and safe path, at acceptable cost — where the unit of evaluation is a whole trajectory of decisions, tool calls, and recoveries rather than a single response. Chapter 9 Safety evals Safety evals systematically probe an AI system's failure modes with adversarial and sensitive inputs — injection, jailbreaks, harmful-content requests, PII handling, bias probes — measuring not whether the system is polite on average but whether it holds at the edges where harm lives. Chapter 10 Cost and latency Cost and latency evals treat speed and spend as first-class quality metrics measured alongside accuracy — per-request and per-outcome, as distributions not averages — because a change that improves answers while doubling cost or p95 latency is not an improvement. Chapter 11 Dashboards Eval dashboards turn scattered eval runs and production quality signals into one continuously visible surface — trends over time, per-slice breakdowns, release annotations — so quality is monitored like uptime instead of rediscovered before each launch. Chapter 12 CI integration CI integration wires evals into the same pipeline that gates code: prompt edits, model version bumps, and retrieval changes trigger eval runs automatically, and failures block merges — making quality regression as hard to ship accidentally as a failing unit test. Chapter 13 Continuous eval Continuous evaluation extends measurement past release into live traffic: sampled production scoring, drift detection against baselines, and feedback loops that recruit real failures into the eval suite — because the world keeps changing after your last deploy. Chapter 14 Eval lab The eval lab is a hands-on exercise to build a small golden evaluation set for a real AI feature, run it to get a baseline, then deliberately introduce a regression and confirm the evaluation actually catches it. Chapter 15 Mini project The mini project ships a small, complete evaluation system for one real AI feature, including a golden dataset, a CI-integrated check, and a basic dashboard, that another engineer could extend to a different feature. Chapter 16 Capstone The capstone designs a complete evaluation strategy for a real AI product: golden dataset design, cost and latency tracking alongside quality, CI integration, continuous production evaluation, and dashboards, as one coherent system. Chapter 17 Interview pack The evaluation interview pack prepares you to discuss AI quality assurance design credibly: golden dataset construction, CI integration, and continuous evaluation for drift, backed by a concrete story rather than a general statement that evaluation matters.
