Evaluation · Chapter 13

Continuous eval

Continuous eval means re-running evaluation regularly against live or recent production data, not just once before launch, since model behavior, user query patterns, and underlying source content can all drift after deployment in ways a one-time evaluation would never catch. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

  • Inputs and outputs: identify what enters Continuous eval, what state changes, and what the next system component receives.
  • Evaluation signal: test Continuous eval with representative success cases, edge cases, and at least one failure case.
  • Operational concern: assign ownership for freshness, permissions, cost, latency, and rollback where this concept touches production.
  • Debugging move: capture traces or artifacts that show whether Continuous eval worked for the right reason.

Common misunderstandings

  • A system that passed its evaluation before launch is not guaranteed to still pass months later; model updates, changing user behavior, and evolving source content can all degrade quality gradually, which only continuous evaluation, not a one-time pre-launch check, can catch.
  • Continuous eval is not complete when the happy path works once; it needs representative evals and visible failure handling.
  • Do not hide Continuous eval inside prompt text when it requires code, policy, data, or operational ownership.

What happens under the hood

The internal flow centers on scheduling the evaluation suite to run on a recurring cadence against recent production queries or a periodically refreshed golden set, tracking the trend over time rather than a single point-in-time score, and alerting when quality trends downward, then hands a validated result to the next step in the product workflow.

Trade-offs

  • Quality vs latency: The main trade-off is continuous evaluation catches drift and degradation that would otherwise go unnoticed for a long time, but running comprehensive evaluations repeatedly has real ongoing compute and maintenance cost, so cadence should match how quickly the specific system's inputs and behavior actually change
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.

Production considerations

Before release, define ownership, SLOs, model fallback rules, data retention, security review criteria, and rollback strategy. For regulated or enterprise environments, add approval gates for prompt changes, model changes, tool permissions, and retrieval corpus updates.
