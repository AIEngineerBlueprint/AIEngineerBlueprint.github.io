AWS AI Stack · Chapter 11

IAM

AWS IAM controls which users, services, and roles can perform which actions on which resources, forming the permission boundary around every other AWS service an AI system touches, from Bedrock model access to S3 document buckets.

Calculating reading timeHands-on labQuiz

In plain terms

It is like a building's keycard system where every door, elevator, and room is individually programmable for exactly who can enter and what they can do once inside.

How it works

In practice it comes down to policies attached to users, groups, or roles that explicitly allow or deny specific actions on specific resources, following least-privilege scoping, with roles assumed temporarily by services like ECS tasks or Lambda functions rather than long-lived credentials embedded in code.

A concrete example

An ECS task running a RAG API assumes an IAM role that can read from one specific S3 prefix and call Bedrock's Converse API, but cannot list all S3 buckets in the account or call unrelated services.

Trade-offs

  • The core tension: Tightly scoped least-privilege roles reduce blast radius if a service is compromised, but they require more upfront design work and more maintenance as services and their needs evolve, compared to broad permissive policies that are easy to set up but risky

Try it in code

{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow", "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::rag-documents/raw/*" },
    { "Effect": "Allow", "Action": "bedrock:InvokeModel",
      "Resource": "arn:aws:bedrock:*::foundation-model/anthropic.claude-3-haiku*" }
  ]
}

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Set up and test IAM in AWS

Make the first win concrete: prove identity, permissions, endpoint reachability, and one expected model or service response before building a full app.

Prerequisites

  • Confirm the target AWS region, account, IAM role, service quota, and billing guardrail before building.
  • Start with a read-only identity check and one minimal service API call.
  • Record the exact resource names, ARNs, regions, and CloudWatch log locations used by the test.

Smoke test

aws sts get-caller-identity
aws configure get region
# Then run the smallest read-only command for the service in this chapter and save the request id.

How to know it worked

  • AccessDenied means IAM or resource policy is the first problem to solve.
  • ResourceNotFound usually means wrong region, wrong name, or missing setup.
  • A useful smoke test records identity, region, request id, latency, and expected output.

Next step: Only after the smoke test passes should you add model calls, data ingestion, write permissions, or automation.

Hands-on lab

Task: Write a least-privilege IAM policy for a Lambda function that only needs to read from one S3 prefix and invoke one specific Bedrock model, denying everything else by default.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Attaching a broad managed administrator policy to get something working quickly is not a free shortcut; overly broad permissions in AI pipelines are a common root cause of real data exposure incidents once forgotten.
