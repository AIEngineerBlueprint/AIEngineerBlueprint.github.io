AWS AI Stack · Chapter 14

AWS lab

The AWS lab is a hands-on exercise to deploy one small AI workload across at least three real AWS services, such as Lambda, S3, and Bedrock, configure IAM roles between them, and verify it end to end with a real request.

Calculating reading timeBuild project

What you're building, and why

In practice it comes down to standing up a minimal but real deployment, such as a Lambda function that reads a document from S3 and calls Bedrock, with IAM roles scoping exactly what each step can access, then testing it with an actual request and checking CloudWatch logs for the trace.

It is like wiring a small working circuit on a breadboard before trusting yourself to wire a whole building.

The brief

Deploy a Lambda function that reads from S3 and calls Bedrock, using a least-privilege IAM role, and verify a real end-to-end request in CloudWatch logs.

A worked reference

A lab might deploy a Lambda function triggered by an S3 upload that calls Bedrock to summarize the new document and writes the result back to a different S3 prefix, verified with a real file upload and a CloudWatch log check.

Set up and test AWS lab in AWS

Make the first win concrete: prove identity, permissions, endpoint reachability, and one expected model or service response before building a full app.

Prerequisites

  • Confirm the target AWS region, account, IAM role, service quota, and billing guardrail before building.
  • Start with a read-only identity check and one minimal service API call.
  • Record the exact resource names, ARNs, regions, and CloudWatch log locations used by the test.

Smoke test

aws sts get-caller-identity
aws configure get region
# Then run the smallest read-only command for the service in this chapter and save the request id.

How to know it worked

  • AccessDenied means IAM or resource policy is the first problem to solve.
  • ResourceNotFound usually means wrong region, wrong name, or missing setup.
  • A useful smoke test records identity, region, request id, latency, and expected output.

Next step: Only after the smoke test passes should you add model calls, data ingestion, write permissions, or automation.

Starting point in code

# Lab: S3 upload -> Lambda -> Bedrock -> S3, verified end to end
aws s3 cp report.pdf s3://rag-documents/raw/report.pdf
aws logs tail /aws/lambda/summarize-on-upload --since 2m
# Pass condition: summary object appears in s3://rag-documents/processed/

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Trade-offs to document

  • The core tension: A real, minimal multi-service deployment teaches IAM and service-integration lessons that reading documentation alone doesn't, but it takes longer to set up than a single-service toy example

Definition of done

Watch out for

  • The lab isn't complete when the Lambda function runs once in the console; it needs a real IAM role, not a wildcard-permission shortcut, and a real CloudWatch trace showing the actual request path.
