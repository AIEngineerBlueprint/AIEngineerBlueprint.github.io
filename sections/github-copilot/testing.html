GitHub Copilot · Chapter 7

Testing

Using Copilot for testing means generating candidate test cases for existing or new code, then verifying that those tests actually assert meaningful behavior and pass in CI, not treating generation itself as proof of coverage.

Calculating reading timeHands-on labQuiz

In plain terms

It is like having someone suggest exam questions for you: useful, but you still have to check the questions actually test the right thing.

How it works

In practice it comes down to generating test cases from an existing function's signature and behavior, or from a natural-language description of expected behavior, then running them to check they actually exercise meaningful edge cases rather than trivially passing.

A concrete example

Copilot suggests three tests for a validation function covering valid, empty, and malformed input, but a developer still checks that the malformed-input test asserts the correct error rather than just calling the function without failing.

Trade-offs

  • The core tension: Generated tests broaden coverage quickly, but a generated test that trivially passes or asserts something too weak gives false confidence, so a human still has to judge test quality, not just test presence

Try it in code

// Generated tests for validate(input) — classify before trusting:
test("valid input passes", () => { expect(validate(good)).toBe(true); });
test("empty input rejected", () => { expect(validate([])).toBe(false); });
test("malformed input rejected", () => {
  expect(() => validate(malformed)).toThrow("invalid shape"); // check the assertion, not just "no crash"
});

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Ask Copilot to generate tests for one function, then classify each generated test as meaningful, redundant, or too weak to catch a real bug.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • A green test suite that Copilot helped generate is not automatically strong coverage; a test that never fails for the right reason provides no real safety net.
