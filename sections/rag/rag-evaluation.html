RAG evaluation

RAG evaluation measures whether retrieval found the right evidence and whether generation used that evidence faithfully. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

Calculating reading timeHands-on labInteractive quiz

Learning objectives

  • Explain why RAG evaluation is important when building AI systems for users, not demos.
  • Describe the internal flow from input signals to model behavior, tool execution, or product outcome.
  • Choose architecture patterns and trade-offs for latency, accuracy, safety, cost, and maintainability.
  • Implement a small production-shaped slice with evaluation, observability, and failure handling.

For non-technical readers

The business-relevant idea here: 'the AI sounded confident' is not the same as 'the AI was right,' and a well-run team should be able to show you an actual test scorecard proving their document-answering assistant finds the correct source and represents it accurately, on a real set of test questions, not just a few times someone tried it and it looked fine. If a team can't show you that kind of evidence before launch, the feature hasn't really been validated yet.

Core concept you must understand

What to learn

  • Core mechanism: golden questions, expected sources, recall@k, faithfulness, citation precision, abstention tests, and regression tracking.
  • Inputs and outputs: identify what enters RAG evaluation, what state changes, and what the next system component receives.
  • Evaluation signal: test RAG evaluation with representative success cases, edge cases, and at least one failure case.
  • Operational concern: assign ownership for freshness, permissions, cost, latency, and rollback where this concept touches production.
  • Debugging move: capture traces or artifacts that show whether RAG evaluation worked for the right reason.

Common misunderstandings

  • Do not use only answer ratings; inspect source recall and citation support separately.
  • RAG evaluation is not complete when the happy path works once; it needs representative evals and visible failure handling.
  • Do not hide RAG evaluation inside prompt text when it requires code, policy, data, or operational ownership.

How to think about it

When to use it: RAG evaluation matters because it changes product quality, reliability, safety, cost, or developer velocity in real AI systems.

When it breaks: Do not use only answer ratings; inspect source recall and citation support separately.

What to ask: What are the inputs, what evidence or data makes the result trustworthy, how will failure be detected, and what should the product do when confidence is low?

Trade-offs

  • Quality vs latency: The main trade-off is Optimizing answer quality alone can hide retrieval failures.
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.

Interactive example

For RAG evaluation, improve evidence before tuning prompts: use better inputs, add targeted checks around flexibility vs reliability, and compare outputs against the chapter exercise or eval set.

Small runnable example

Use this snippet as the smallest concrete anchor for the concept. Change the inputs, then observe which assumptions fail.

async function evaluateRetrieval(query, expectedSourceId) {
  const results = await retrieve(query, { topK: 5 });
  return {
    foundExpectedSource: results.some((r) => r.sourceId === expectedSourceId),
    topScore: results[0]?.score ?? 0,
    returnedSources: results.map((r) => r.sourceId)
  };
}

Production considerations

Before release, define ownership, SLOs, model fallback rules, data retention, security review criteria, and rollback strategy. For regulated or enterprise environments, add approval gates for prompt changes, model changes, tool permissions, and retrieval corpus updates.
