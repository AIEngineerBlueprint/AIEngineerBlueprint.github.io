Retrieval

Retrieval is the step in RAG that finds the most relevant chunks for a given query by comparing the query's embedding to stored chunk embeddings and returning the top K nearest neighbours.

Calculating reading timeHands-on labQuiz

When it goes wrong

The right document is in the index — at position 27 of the ranked results. Your prompt only includes the top 5. Retrieval quality, not the LLM, is why the answer is wrong.

Why this matters

Retrieval quality is the single most important factor in RAG answer quality. A perfect generator with bad retrieval produces wrong answers.

In plain terms

Retrieval is like a librarian who has read all the index cards and, the moment you ask a question, instantly hands you the most relevant ones without reading every card each time.

How it works

Internal flow: user query → embed query → vector similarity search → top K chunks → (re-rank) → pass to LLM with original query.

  • Dense retrieval: embed the query with the same model used to embed chunks. Find nearest neighbours in vector space using cosine similarity or dot product.
  • Sparse retrieval (BM25): match query terms directly against document terms with TF-IDF weighting. Better for exact keyword matches and rare terms.
  • Hybrid retrieval: combine dense and sparse scores. Usually outperforms either alone, especially for queries mixing semantic intent and specific keywords.
  • Top-K: retrieve the K most similar chunks. K is a hyperparameter — too small misses relevant content, too large floods the context with noise.
  • Query rewriting: paraphrase or expand the user's query before retrieval. Improves recall when user queries are ambiguous or short.

A concrete example

User asks: 'What is our Q3 refund policy for enterprise customers?' Dense retrieval finds chunks about refunds semantically. BM25 finds chunks containing 'Q3' and 'enterprise'. Hybrid retrieval combines both signals and returns the specific policy section more reliably than either alone.

Trade-offs

  • The core tension: Dense retrieval is better for semantic similarity; BM25 is better for exact terms; hybrid is better overall but adds complexity and latency.

Try it in code

async function retrieve(query, embed, vectorDb) {
  const queryVector = await embed(query);
  const matches = await vectorDb.search(queryVector, { topK: 5 });
  return matches.filter((match) => match.score >= 0.78);
}

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Take 10 questions for your use case and manually find the correct chunk in your document corpus. Then run your retrieval system. For each question, did the correct chunk appear in the top 3, top 5, or not at all? This is your recall@3 and recall@5.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

To make it production-shaped: Eval: measure retrieval recall@K — what fraction of questions have the correct chunk in the top K results. Fix retrieval before tuning the generator.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Vector similarity is not the same as relevance. High cosine similarity does not guarantee the chunk actually answers the query.
  • More retrieved chunks are not always better. Irrelevant chunks in the context confuse the generator and increase cost.
  • Retrieval and generation failures have different signatures — wrong answer vs no answer. Diagnose which is failing before tuning.
