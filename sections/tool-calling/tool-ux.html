Tool Calling · Chapter 11

Tool UX

Tool UX is how a tool call and its result are surfaced to a human user, showing what action is being taken, on what data, and with what outcome, so users can understand and trust what an agent is doing on their behalf.

Calculating reading timeHands-on labQuiz

In plain terms

It is like a food delivery app showing each stage, order placed, preparing, out for delivery, rather than a silent wait with no visibility into what's happening.

How it works

In practice it comes down to displaying a human-readable description of the tool being called and its purpose, showing the specific arguments or data involved, and surfacing the result clearly, rather than only showing a generic loading state or a final answer with no visibility into intermediate actions.

A concrete example

An AI assistant that searches a calendar and books a meeting shows the user a message like checking your calendar for Tuesday afternoon, found a slot at 2pm, booking now, rather than a silent pause followed by a sudden confirmation.

Trade-offs

  • The core tension: Showing detailed tool-call activity builds user trust and understanding, but excessive detail on every minor tool call can clutter the interface and slow down the perceived experience for simple tasks

Try it in code

<tool-activity>
  <status>Checking your calendar for Tuesday afternoon...</status>
  <status>Found a slot at 2pm, booking now...</status>
  <result>✓ Meeting booked for Tue 2:00pm</result>
</tool-activity>

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: For one real agentic feature, design how you would surface one specific tool call, its purpose, and its result to the end user.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Hiding tool calls from the user is not automatically a cleaner experience; users tend to trust and understand an AI agent more when they can see what actions it's taking on their behalf, especially for anything with a real-world effect.
