Tool Calling · Chapter 14

Tool lab

The tool lab is a hands-on exercise to implement one real tool with a schema, argument validation, and structured error handling, then test it against both a valid call and a deliberately invalid one.

Calculating reading timeBuild project

What you're building, and why

In practice it comes down to defining a tool schema, implementing argument validation and structured error responses, and testing the tool against a valid call and an invalid call to confirm both the happy path and the error path behave as designed.

It is like building and testing one real appliance's safety features, not just plugging it in and hoping it works.

The brief

Implement one real tool with a schema and validation, then test it with both a valid call and a deliberately invalid one, confirming the error response is helpful.

A worked reference

A lab might implement a book_meeting tool with a schema requiring a time and attendee list, validate that the time isn't in the past, and test both a valid booking and an attempt to book a meeting in the past to confirm the structured error fires correctly.

Starting point in code

// Valid call
await bookMeeting({ time: "2025-06-01T14:00:00Z", attendees: ["a@co.com"] });
// Invalid call — time in the past
await bookMeeting({ time: "2020-01-01T14:00:00Z", attendees: ["a@co.com"] });
// Expect: structured error, not a silent failure or a booked meeting

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Trade-offs to document

  • The core tension: Testing both valid and invalid calls takes more setup than just confirming the tool works on a clean example, but it's the only way to know the validation and error handling actually function as designed

Definition of done

Watch out for

  • The lab is not complete once a tool works on a valid call; without testing the invalid case and confirming a helpful structured error, half the tool's real design isn't actually verified.
