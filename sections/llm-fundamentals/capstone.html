LLM Fundamentals · Chapter 22

Capstone

The capstone designs a complete LLM-powered system for a real use case: model selection justified by task fit, an inference architecture accounting for context and caching, sampling strategy, and a plan for the trade-offs between latency, cost, and quality. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

What to learn

  • Core mechanism: combining a model choice justified against the task's specific requirements, an inference architecture that accounts for context window limits and caching opportunities, a deliberate sampling strategy matched to the task's need for consistency or creativity, and an explicit accounting of the latency-cost-quality trade-off into one complete, documented system design.
  • Inputs and outputs: identify what enters Capstone, what state changes, and what the next system component receives.
  • Evaluation signal: test Capstone with representative success cases, edge cases, and at least one failure case.
  • Operational concern: assign ownership for freshness, permissions, cost, latency, and rollback where this concept touches production.
  • Debugging move: capture traces or artifacts that show whether Capstone worked for the right reason.

Common misunderstandings

  • An LLM-powered feature is not fully designed once it produces reasonable output in testing; without a deliberate accounting of the latency, cost, and quality trade-offs behind the model and sampling choices, the design hasn't actually been thought through, it's just working by default.
  • Capstone is not complete when the happy path works once; it needs representative evals and visible failure handling.
  • Do not hide Capstone inside prompt text when it requires code, policy, data, or operational ownership.

What to ask: What are the inputs, what evidence or data makes the result trustworthy, how will failure be detected, and what should the product do when confidence is low?

What happens under the hood

Trade-offs

  • Quality vs latency: The main trade-off is a complete LLM system design spanning model choice, inference architecture, and sampling strategy takes substantially more thought than simply calling a default model with default settings, but it reflects what's actually required to build a genuinely well-considered LLM-powered product
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.

Production considerations

Before release, define ownership, SLOs, model fallback rules, data retention, security review criteria, and rollback strategy. For regulated or enterprise environments, add approval gates for prompt changes, model changes, tool permissions, and retrieval corpus updates.
