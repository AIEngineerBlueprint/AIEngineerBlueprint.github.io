Learning track

AI Security

A production-focused path for mastering AI Security through concepts, diagrams, hands-on labs, interview practice, and capstone work.

Track outcomes

By the end of this track, you should be able to reason from first principles, design the architecture, identify failure modes, build a working slice, and explain how the system behaves in production.

In plain words: AI security is zero-trust engineering for probabilistic software. Treat prompts, documents, tools, plugins, and model output as untrusted until checked. AI products connect natural language to data and actions, which creates new paths for injection, leakage, and privilege misuse.

Production build path

Use this track to contribute to one of the platform projects below. These projects are intentionally track-level because they combine multiple chapters into a complete product slice.

Enterprise RAG

Build a private documentation assistant with ingestion, chunking, hybrid search, citations, evals, access control, and freshness monitoring.

AI code reviewer

Build a reviewer that reads diffs, classifies risk, checks security rules, and produces actionable comments with confidence levels.

Document intelligence

Extract structured data from PDFs or forms, validate against schemas, route exceptions to humans, and track extraction quality.

AI support bot

Answer customer questions with grounded knowledge, escalation rules, conversation memory, and deflection/CSAT metrics.

Engineering assistant

Connect repo search, issue context, runbooks, and deployment checks into a tool-using assistant for engineers.

Chapter 1 OWASP LLM Top 10 The OWASP LLM Top 10 is a community-maintained list of the ten most critical security vulnerabilities in LLM-based applications, providing a common vocabulary and prioritisation framework for AI security. Chapter 2 Prompt injection Prompt injection is an attack where an adversary embeds instructions into data the model processes (user input, retrieved documents, tool results) that override or manipulate the original system instructions. Chapter 3 Data leakage Data leakage happens when private, sensitive, or evaluation-only data appears where it should not, such as prompts, logs, outputs, training data, or test sets. Chapter 4 Model theft Model theft is extracting model behavior, weights, prompts, or proprietary outputs through attacks, scraping, or misuse. Chapter 5 Supply chain AI supply-chain risk comes from models, datasets, packages, containers, prompts, and tools that may be compromised or untrusted. Chapter 6 Model poisoning Model poisoning manipulates training or fine-tuning data so a model learns hidden malicious behavior or degraded performance. Chapter 7 RAG attacks RAG attacks target the retrieval and grounding layer with poisoned documents, hidden instructions, misleading citations, or permission bypasses. Chapter 8 MCP attacks MCP attacks abuse Model Context Protocol tools, resources, or servers to leak data, escalate permissions, or manipulate agent behavior. Chapter 9 Agent attacks Agent attacks exploit autonomous planning, tool access, memory, or approvals to make agents take harmful actions. Chapter 10 Secrets Secrets are credentials, tokens, keys, or private material that must never be exposed to models, logs, users, or untrusted tools. Chapter 11 Identity Identity controls who the user, agent, tool, and service are so permissions can be enforced correctly. Chapter 12 Sandboxing Sandboxing isolates AI-generated code, tools, or file operations so failures and attacks cannot escape into sensitive systems. Chapter 13 Threat models Threat models identify assets, actors, entry points, trust boundaries, abuses, and mitigations before building or shipping an AI system. Chapter 14 Security lab The security lab is a hands-on exercise to attempt one real prompt-injection or jailbreak technique against an AI system you have access to, observe whether it succeeds, and then implement and verify a specific defense against it. Chapter 15 Mini project The mini project ships a small, complete security review for one real AI feature, covering at least three attack categories from the OWASP LLM Top 10, with a specific tested mitigation for each, that another engineer could use as a template. Chapter 16 Capstone The capstone designs a complete AI security program for a real product: threat model, specific defenses mapped to the OWASP LLM Top 10 categories relevant to the system, identity and sandboxing design, and an ongoing security testing cadence. Chapter 17 Interview pack The AI security interview pack prepares you to discuss LLM-specific threat modeling and defense design credibly: prompt injection, data leakage, and tool-access security, backed by a concrete story about a real vulnerability or defense you implemented.
