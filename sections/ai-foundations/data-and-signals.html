Data and signals

Data and signals refers to the raw information an AI system learns from or acts on, and the specific measurable indicators extracted from that data that the system actually uses to make decisions. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

Learning objectives

  • Explain why Data and signals is important when building AI systems for users, not demos.
  • Describe the internal flow from input signals to model behavior, tool execution, or product outcome.
  • Choose architecture patterns and trade-offs for latency, accuracy, safety, cost, and maintainability.
  • Implement a small production-shaped slice with evaluation, observability, and failure handling.

For non-technical readers

In plain terms: an AI system can only be as smart as the information it's actually shown, no more. If you're evaluating an AI proposal, the most useful non-technical question is what data would this actually learn from, and do we really have that data, in good shape, today. A shockingly common reason AI projects stall isn't the model, it's discovering six months in that the data everyone assumed existed was incomplete, inconsistent, or locked in a system nobody accounted for.

Beginner explanation

If you have no AI background, start here: Data and signals refers to the raw information an AI system learns from or acts on, and the specific measurable indicators extracted from that data that the system actually uses to make decisions.

Analogy: It is like the difference between a company's raw sales records and the specific dashboard metrics, like conversion rate or churn, that leadership actually watches to make decisions; the metrics are a distilled signal extracted from the raw data.

Common misunderstandings

What happens under the hood

Trade-offs

  • Quality vs latency: The main trade-off is using more signals can improve a model's accuracy, but each additional data source adds collection cost, privacy exposure, and potential noise, so signal selection is a real trade-off, not simply more data is always better
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.

Small runnable example

Use this snippet as the smallest concrete anchor for the concept. Change the inputs, then observe which assumptions fail.

function runLearningSlice(input) {
  const result = processInput(input);
  const checks = {
    hasExpectedShape: Boolean(result),
    isObservable: Boolean(result.traceId),
    hasFailurePath: Boolean(result.error || result.value)
  };
  return { result, checks };
}
