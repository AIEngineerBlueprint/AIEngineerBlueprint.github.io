Data and signals

Data and signals refers to the raw information an AI system learns from or acts on, and the specific measurable indicators extracted from that data that the system actually uses to make decisions.

Calculating reading timeHands-on labQuiz

In plain terms

It is like the difference between a company's raw sales records and the specific dashboard metrics, like conversion rate or churn, that leadership actually watches to make decisions; the metrics are a distilled signal extracted from the raw data.

For non-technical readers

In plain terms: an AI system can only be as smart as the information it's actually shown, no more. If you're evaluating an AI proposal, the most useful non-technical question is what data would this actually learn from, and do we really have that data, in good shape, today. A shockingly common reason AI projects stall isn't the model, it's discovering six months in that the data everyone assumed existed was incomplete, inconsistent, or locked in a system nobody accounted for.

How it works

In practice it comes down to identifying what raw data is available, what signals can be reliably extracted from it, and how representative and clean that data is, since a model's behavior is fundamentally shaped by the data and signals it was trained or operated on, not by intent alone.

A concrete example

A churn-prediction model is only as good as the signals available to it; if the data doesn't capture a key reason customers actually leave, like a competitor's new pricing, no amount of modeling skill will let it predict that kind of churn.

Trade-offs

  • The core tension: Using more signals can improve a model's accuracy, but each additional data source adds collection cost, privacy exposure, and potential noise, so signal selection is a real trade-off, not simply more data is always better

Hands-on lab

Task: For one real business decision you'd want an AI system to help with, list what data is actually available, and identify one signal that might be missing that would matter a lot.

  1. Build or sketch the smallest version that actually runs or can be reviewed.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Having a lot of data does not automatically mean having the right signals; a company can have millions of records and still lack the specific signal that actually predicts the outcome it cares about.
