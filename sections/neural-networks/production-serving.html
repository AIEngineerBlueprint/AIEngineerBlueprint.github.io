Neural Networks · Chapter 12

Production serving

Production serving of neural networks is the infrastructure and practices for running a trained model reliably against live traffic, covering batching, hardware allocation, versioning, and fallback behavior when the model or its infrastructure fails.

Calculating reading timeHands-on labQuiz

In plain terms

It is like the difference between a chef who can cook one great meal in a home kitchen, versus the full kitchen operation needed to serve that same meal reliably to hundreds of customers a night.

How it works

In practice it comes down to deploying a trained model behind a serving layer that batches concurrent requests for hardware efficiency, monitors latency and error rates, supports versioned rollouts so a new model can be tested against a fraction of traffic before full rollout, and defines a fallback behavior if the serving infrastructure fails.

A concrete example

A recommendation model is deployed with canary rollout, sending 5 percent of traffic to a newly trained version and comparing its business metrics against the existing production version before gradually increasing its traffic share.

Trade-offs

  • The core tension: Robust serving infrastructure, canary rollouts, monitoring, fallbacks, reduces the risk of a bad model update causing a widespread production incident, but it adds real infrastructure complexity and engineering time compared to simply swapping a model file directly into production

Try it in code

const canaryRollout = {
  newModelTrafficPercent: 5,
  monitor: "business metric vs. current production model",
  rollbackTrigger: "metric regression beyond 2%"
};

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Design a canary rollout plan for deploying one new model version to production: what percentage of traffic you'd start with, what metric you'd monitor, and what would trigger a rollback.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • A model that performs well in offline evaluation is not guaranteed to behave the same in production serving; differences in real-time input distributions, latency constraints, and infrastructure can all cause a model to underperform its offline metrics once actually serving live traffic.
