Building Products · Chapter 8

Evaluation

Evaluation in the context of building AI products means deciding, at the product level, what quality actually means for this specific feature, accuracy, tone, safety, latency, and building the measurement process that proves whether the product meets that bar before and after every change.

Calculating reading timeHands-on labQuiz

In plain terms

It is like a product manager defining what success actually looks like for a feature before engineering starts building it, rather than discovering the definition of success only after launch.

How it works

In practice it comes down to translating product requirements into specific, measurable evaluation criteria, building a representative test set reflecting real user needs, and using that evaluation both as a pre-launch gate and an ongoing production quality signal, tied back to product decisions rather than treated as a purely engineering concern.

A concrete example

A product team defines that a summarization feature must preserve all numeric facts and stay under 100 words, translates that into an automated evaluation check, and uses failing evaluation results as a real go or no-go gate for launch, not just an engineering nice-to-have.

Trade-offs

  • The core tension: Product-level evaluation criteria that reflect real user needs produce more meaningful quality signals than generic technical metrics alone, but defining and maintaining evaluation criteria this specific requires ongoing collaboration between product and engineering, not a one-time engineering task

Try it in code

const launchGate = {
  requirement: "summaries preserve all numeric facts, stay under 100 words",
  evalCheck: (output) => preservesNumbers(output) && wordCount(output) < 100,
  gate: "no launch until 95% of golden set passes"
};

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: For one real AI product feature, translate its product requirements into three specific, measurable evaluation criteria that would function as a launch gate.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Evaluation is not solely an engineering concern to be handled after a feature is built; product decisions about what quality actually means for users need to shape the evaluation criteria from the start, or engineering ends up measuring something that doesn't reflect real product success.
