Evaluation

Evaluation in the context of building AI products means deciding, at the product level, what quality actually means for this specific feature, accuracy, tone, safety, latency, and building the measurement process that proves whether the product meets that bar before and after every change. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

What to learn

  • Inputs and outputs: identify what enters Evaluation, what state changes, and what the next system component receives.
  • Evaluation signal: test Evaluation with representative success cases, edge cases, and at least one failure case.
  • Operational concern: assign ownership for freshness, permissions, cost, latency, and rollback where this concept touches production.
  • Debugging move: capture traces or artifacts that show whether Evaluation worked for the right reason.

Common misunderstandings

  • Evaluation is not solely an engineering concern to be handled after a feature is built; product decisions about what quality actually means for users need to shape the evaluation criteria from the start, or engineering ends up measuring something that doesn't reflect real product success.
  • Evaluation is not complete when the happy path works once; it needs representative evals and visible failure handling.
  • Do not hide Evaluation inside prompt text when it requires code, policy, data, or operational ownership.

How to think about it

When to use it: Evaluation matters because it changes product quality, reliability, safety, cost, or developer velocity in real AI systems.

When it breaks: Evaluation is not solely an engineering concern to be handled after a feature is built; product decisions about what quality actually means for users need to shape the evaluation criteria from the start, or engineering ends up measuring something that doesn't reflect real product success.

What to ask: What are the inputs, what evidence or data makes the result trustworthy, how will failure be detected, and what should the product do when confidence is low?

What happens under the hood

The internal flow centers on translating product requirements into specific, measurable evaluation criteria, building a representative test set reflecting real user needs, and using that evaluation both as a pre-launch gate and an ongoing production quality signal, tied back to product decisions rather than treated as a purely engineering concern, then hands a validated result to the next step in the product workflow.

Trade-offs

  • Quality vs latency: The main trade-off is product-level evaluation criteria that reflect real user needs produce more meaningful quality signals than generic technical metrics alone, but defining and maintaining evaluation criteria this specific requires ongoing collaboration between product and engineering, not a one-time engineering task
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.

Small runnable example

Use this snippet as the smallest concrete anchor for the concept. Change the inputs, then observe which assumptions fail.

  requirement: "summaries preserve all numeric facts, stay under 100 words",
  evalCheck: (output) => preservesNumbers(output) && wordCount(output) < 100,
  gate: "no launch until 95% of golden set passes"

Production considerations

Before release, define ownership, SLOs, model fallback rules, data retention, security review criteria, and rollback strategy. For regulated or enterprise environments, add approval gates for prompt changes, model changes, tool permissions, and retrieval corpus updates.
