Building Products · Chapter 3

Architecture

Architecture for an AI product is the overall system design connecting the user-facing application, the model or model gateway, retrieval and tool integrations, guardrails, and observability into one coherent, deployable system, distinct from any single component's internal design.

Calculating reading timeHands-on labQuiz

In plain terms

It is like the blueprint for an entire building, showing how plumbing, electrical, and structural systems connect, rather than the detailed design of any one room.

How it works

In practice it comes down to defining how a request flows from the user interface through authentication, any retrieval or tool calls, the model invocation, guardrail checks, and back to the user, with clear boundaries and contracts between each component so they can be built, tested, and scaled somewhat independently.

A concrete example

A customer support product's architecture routes a user message through an auth check, a retrieval step against a knowledge base, a guardrail-wrapped model call, and a logging step, with each stage owned by a clear contract so the retrieval system can be upgraded without touching the guardrail logic.

Trade-offs

  • The core tension: A well-decoupled architecture makes individual components easier to test, scale, and replace independently, but designing clean boundaries between components takes more upfront design effort than building one tightly coupled system quickly

Try it in code

const requestFlow = [
  "auth check",
  "retrieval (knowledge base)",
  "guardrail-wrapped model call",
  "logging + trace ID",
  "response to user"
];

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Sketch the architecture for one real AI product, showing the request flow from user input through every major component to the final response, and identify one boundary that should be a clean, independent contract.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • An AI product's architecture is not just which model API you call; the retrieval, guardrail, and observability components around the model call are what actually determine whether the product is reliable, safe, and debuggable in production.
