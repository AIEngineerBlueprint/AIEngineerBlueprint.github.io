Structured Output · Chapter 9

Workflows

Workflows for structured output chain multiple structured-output steps together, where one step's validated output becomes the next step's input, letting complex multi-stage processes stay reliable end to end.

Calculating reading timeHands-on labQuiz

In plain terms

It is like an assembly line where each station's output must meet a specification before it's passed to the next station, rather than passing along whatever came out regardless of whether it's correct.

How it works

In practice it comes down to designing each stage of a multi-step process to produce schema-validated output, checking that output before passing it to the next stage, so an error at one stage is caught early rather than silently propagating and compounding through the rest of the workflow.

A concrete example

A document-processing workflow extracts structured fields in one step, validates them, then uses only the validated fields as input to a second step that generates a structured summary, rather than passing raw, unvalidated extraction results forward.

Trade-offs

  • The core tension: Validating structured output at each stage of a workflow catches errors early and prevents them from compounding, but it adds a validation step at every stage, which takes more design effort than a single end-to-end unstructured pass

Try it in code

const fields = await extractStructured(doc, invoiceSchema);
if (!isValid(fields, invoiceSchema)) throw new StageError("extraction");
const summary = await generateStructured(fields, summarySchema); // only validated input passed forward

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: For one real multi-step process, identify one intermediate stage where validating structured output before passing it forward would prevent an error from compounding.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • A multi-step workflow doesn't need validation only at the final output; an error introduced early in a chain of structured steps can compound and become harder to trace the further it propagates, so each stage's output should be validated before it feeds the next.
