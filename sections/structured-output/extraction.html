Structured Output · Chapter 7

Extraction

Extraction is using a model to pull specific structured fields out of unstructured text, such as pulling a date, amount, and vendor name from an invoice, constrained to a defined schema so the output is directly usable by downstream code.

Calculating reading timeHands-on labQuiz

In plain terms

It is like a skilled clerk reading a messy handwritten form and filling out a clean, standardized digital form with exactly the fields that matter.

How it works

In practice it comes down to prompting a model with the source text and a target schema, constraining its output to that schema, and validating the extracted fields against expected types and formats before passing them to downstream systems.

A concrete example

An extraction system pulls invoice number, due date, and total amount from scanned invoices with varying layouts, producing consistent structured records regardless of each invoice's specific visual format.

Trade-offs

  • The core tension: Extraction handles varied, messy input formats far more flexibly than a rules-based parser, but it can occasionally misextract a field on an unusual layout, so validation and a human-review path for low-confidence extractions still matter

Try it in code

const schema = { invoiceNumber: "string", dueDate: "string", totalAmount: "number" };
const extracted = await extractStructured(invoiceText, schema);
validateAgainstSchema(extracted, schema); // don't trust it just because it parsed

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: For one real unstructured document type, design an extraction schema listing the specific fields you'd want pulled out.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • A model successfully extracting fields on most documents does not guarantee it will extract correctly on a genuinely novel layout; extraction systems still need validation and a fallback for low-confidence or malformed results.
