Refactoring

Refactoring with Claude Code means using it to restructure code without changing behavior, verified by running the existing test suite before and after the change, not just by reading the diff.

Calculating reading timeHands-on labQuiz

In plain terms

It is like reorganizing a warehouse's shelving without moving any of the products to the wrong section.

How it works

In practice it comes down to scoping a refactor to an explicit, bounded set of files and behavior-preserving goals, then asking the agent to make the change and run tests before and after to confirm behavior did not shift.

A concrete example

Extracting a shared validation function used in four places works well when the agent is told the exact four call sites and the test command to confirm nothing else changed; an unscoped request risks touching unrelated code.

Trade-offs

  • The core tension: A tightly scoped refactor request is easy to review and verify, while a broad request produces a larger diff that is harder to check line by line and more likely to include unintended behavior changes

Try it in code

claude "extract the duplicated email-validation logic in
signup.ts, login.ts, and resetPassword.ts into one shared
function in validators.ts. Only touch these four files.
Run npm test before and after to confirm behavior is unchanged."

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Pick a function with duplicated logic in three places, ask the agent to extract a shared helper, and verify test results are identical before and after.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • A refactor that compiles and looks cleaner is not automatically safe; without a passing test suite covering the affected behavior, a refactor can silently change edge-case behavior.
