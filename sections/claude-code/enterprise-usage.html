Claude Code · Chapter 10

Enterprise usage

Enterprise usage of Claude Code involves organization-wide managed settings, centralized permission policy, and audit logging so security and platform teams can enforce consistent rules across every developer's agent, rather than relying on each developer configuring safety individually.

Calculating reading timeHands-on labQuiz

In plain terms

It is like a company-wide security policy that applies the same way to every laptop, instead of trusting each employee to configure their own firewall.

How it works

In practice it comes down to centrally managed policy settings that IT or security teams push to every developer's machine, controlling allowed tools, MCP servers, and model access, layered underneath and unable to be overridden by individual or project-level settings.

A concrete example

A platform team ships a managed settings file that blocks all agents from running curl to external hosts by default and requires an approved MCP server allowlist, applied the same way whether a developer is new or senior.

Trade-offs

  • The core tension: Centralized policy gives security teams confidence and consistency across the org, but overly restrictive managed settings can block legitimate workflows and push developers toward workarounds if the policy is not tuned with real usage data

Try it in code

// Managed settings (deployed by IT/security, cannot be
// overridden by project or user settings)
{
  "permissions": {
    "deny": ["Bash(curl*)", "Bash(wget*)"]
  },
  "mcpServers": {
    "allowedServers": ["internal-docs", "jira"]
  }
}

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Draft a managed-policy settings file for a hypothetical fintech engineering org: list three tool categories to deny by default and one MCP server to allow, with a one-sentence justification for each.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Enterprise policy is not just documentation that developers are asked to follow; managed settings are enforced at the tool level and take precedence over a developer's local configuration.
