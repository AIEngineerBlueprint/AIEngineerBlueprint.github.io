Memory lab

The memory lab is a hands-on exercise to build and verify a minimal memory system: store, retrieve, update, and forget, each demonstrated with visible input and output rather than left as theory.

Calculating reading timeBuild project

What you're building, and why

In practice it comes down to building a minimal working memory system end to end that stores a few facts, retrieves relevant ones for a new query, forgets one on request, and verifies each step with a visible before-and-after state.

It is like assembling a small working filing system by hand so you understand every part before trusting a larger automated one.

The brief

Build the four-step lab of store, retrieve, update, and forget for three sample facts, and write down what broke first when you added a fourth, conflicting fact.

A worked reference

A lab implementation stores three facts about a fictional user, retrieves the relevant one for a sample query, updates one when it changes, and deletes one on a forget request, verifying each state change.

Starting point in code

// Lab: store, retrieve, update, forget — verify each transition.
await store(userId, { key: "editor", value: "VS Code" });
console.log(await retrieve(userId, "what editor do I use"));
await update(userId, "editor", "Neovim");
await forget(userId, "editor");
console.log(await retrieve(userId, "what editor do I use")); // must be empty

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Trade-offs to document

  • The core tension: A minimal working implementation teaches the real mechanics faster than reading about memory architecture, but it will oversimplify concerns like access control and retrieval ranking that matter at scale

Definition of done

Watch out for

  • The lab is not complete when storage and retrieval work for the happy path; forgetting and conflict handling are the parts that actually distinguish a real memory system from a lookup table.
