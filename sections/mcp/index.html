Learning track

MCP

A production-focused path for mastering MCP through concepts, diagrams, hands-on labs, interview practice, and capstone work.

Track outcomes

By the end of this track, you should be able to reason from first principles, design the architecture, identify failure modes, build a working slice, and explain how the system behaves in production.

In plain words: MCP is like a USB-C port for AI applications: one standard way for a model client to discover tools, resources, and prompts from many servers. A protocol boundary keeps tool access governable. Teams can add capabilities without hard-wiring every integration into every assistant.

Production build path

Use this track to contribute to one of the platform projects below. These projects are intentionally track-level because they combine multiple chapters into a complete product slice.

Enterprise RAG

Build a private documentation assistant with ingestion, chunking, hybrid search, citations, evals, access control, and freshness monitoring.

AI code reviewer

Build a reviewer that reads diffs, classifies risk, checks security rules, and produces actionable comments with confidence levels.

Document intelligence

Extract structured data from PDFs or forms, validate against schemas, route exceptions to humans, and track extraction quality.

AI support bot

Answer customer questions with grounded knowledge, escalation rules, conversation memory, and deflection/CSAT metrics.

Engineering assistant

Connect repo search, issue context, runbooks, and deployment checks into a tool-using assistant for engineers.

Chapter 1 Protocol overview MCP protocol overview explains how clients, servers, tools, resources, and prompts exchange structured context so AI apps can connect to external capabilities safely. Chapter 2 MCP architecture MCP architecture defines where the host app, client, server, tools, resources, prompts, permissions, and transport boundaries sit. Chapter 3 Transports Transports in MCP are the communication mechanisms, such as stdio for local processes or HTTP-based streaming for remote servers, that carry messages between an MCP client and server, chosen based on whether the server runs locally or remotely. Chapter 4 Resources Resources in MCP are a way for a server to expose readable data, such as files or database records, to a client in a structured, addressable way, distinct from tools which perform actions. Chapter 5 Tools Tools in MCP are the functions a server exposes that a client's model can invoke to perform an action or computation, following the same schema-based function-calling pattern used across most agentic AI systems. Chapter 6 Prompts Prompts in MCP are reusable, server-defined prompt templates that a client can discover and use, letting a server share proven prompting patterns for its specific domain rather than every client having to write its own from scratch. Chapter 7 Server implementation MCP server implementation exposes tools, resources, and prompts through a protocol contract that clients can discover and call. Chapter 8 Client integration MCP client integration connects an AI host to MCP servers, discovers capabilities, chooses tools/resources, and enforces user or workspace policy. Chapter 9 Security model The MCP security model defines how servers, tools, resources, users, and hosts are trusted, permissioned, isolated, and audited. Chapter 10 Enterprise governance Enterprise governance for MCP defines how organizations approve servers, manage versions, enforce policies, monitor usage, and respond to incidents. Chapter 11 Observability Observability gives engineers enough traces, metrics, logs, and examples to understand AI system behavior in production. Chapter 12 Versioning Versioning for MCP means managing changes to a server's exposed tools, resources, and prompts over time so that connected clients aren't unexpectedly broken by a change to the server's interface. Chapter 13 Operations Operations for MCP means monitoring server uptime, tool call latency and error rate, and resource access patterns in production, so a connected agent's failures can be traced back to the specific MCP server or tool responsible. Chapter 14 MCP lab The MCP lab is a hands-on exercise to build a small MCP server exposing one resource and one tool, connect it to a client, and verify both work correctly end to end. Chapter 15 Mini project The mini project ships a small, complete MCP server for one real use case, with a documented resource, a validated tool, and basic operational monitoring, that another engineer could connect a client to. Chapter 16 Capstone The capstone designs a complete MCP server architecture for a real system: resource and tool design, transport choice, versioning strategy, security model, and operational monitoring, as one coherent system. Chapter 17 Interview pack The MCP interview pack prepares you to discuss protocol design choices credibly: resources versus tools, transport selection, and versioning strategy, backed by a concrete story rather than a definition of the protocol.
