Distributed training

Distributed training splits a model's training workload across multiple GPUs or machines, either by dividing the data, data parallelism, or dividing the model itself, model parallelism, to train faster or handle models too large for a single device.

Calculating reading timeHands-on labQuiz

In plain terms

It is like a large printing job split across multiple printers running simultaneously, versus one printer handling the entire job alone, either splitting the pages, data parallelism, or splitting a single oversized document across machines, model parallelism.

How it works

In practice it comes down to data parallelism running the same model on multiple devices with different data batches and synchronizing gradients afterward, while model parallelism splits the model's layers or components across devices when the model itself is too large to fit on one device, each approach with different communication overhead.

A concrete example

A large model too big to fit in a single GPU's memory is split across four GPUs, with each holding a portion of the model's layers, while a smaller model that fits easily is instead replicated across the same four GPUs with each processing a different data batch.

Trade-offs

  • The core tension: Distributing training speeds up large training runs and enables training models too big for one device, but coordinating multiple devices adds real communication overhead and engineering complexity that isn't worth it for models that train fast enough on a single device

Try it in code

// Data parallelism: same model, different data shards
const gradients = await Promise.all(
  shards.map(shard => worker.computeGradients(model, shard))
);
model.applyGradients(averageGradients(gradients));

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: For one real or hypothetical large model, decide whether it would need model parallelism, data parallelism, or neither, and justify the choice based on whether it fits on a single device.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Adding more GPUs to a training run does not always produce a proportional speedup; communication overhead between devices can eat into the gains, especially if the workload isn't large enough to justify the coordination cost.
