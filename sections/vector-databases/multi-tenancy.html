Vector Databases · Chapter 10

Multi-tenancy

Multi-tenancy in a vector database means isolating one customer or team's vectors from another's, ensuring a similarity search never returns or is influenced by data belonging to a different tenant.

Calculating reading timeHands-on labQuiz

In plain terms

It is like separate locked storage units in the same facility: everyone shares the building, but no unit's contents are visible or accessible from another.

For non-technical readers

If your company sells software to multiple customers, this is the specific technical guarantee that keeps one customer's data from ever leaking into another customer's AI search results. It's exactly the kind of thing worth asking a vendor about directly: 'how do you guarantee my company's data never shows up in another customer's results,' since a vague answer here is a real red flag for a multi-customer product handling any sensitive data.

How it works

In practice it comes down to partitioning vectors by tenant through separate indexes, separate namespaces within a shared index, or a mandatory tenant-ID metadata filter enforced on every single query, chosen based on tenant count, isolation requirements, and query performance needs.

A concrete example

A SaaS product serving thousands of customers uses a shared index with a mandatory tenant-ID filter applied to every query at the infrastructure level, rather than trusting application code to always remember to add the filter.

Trade-offs

  • The core tension: Separate indexes per tenant give the strongest isolation guarantee and simplest security reasoning, but they don't scale well to thousands of small tenants, while a shared index with enforced filtering scales better but requires the filter enforcement to be bulletproof, not just a convention

Try it in code

async function search(tenantId, queryVector, topK = 5) {
  return index.query({
    vector: queryVector,
    topK,
    filter: { tenantId: { "$eq": tenantId } } // enforced, not optional
  });
}

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: For one real multi-tenant system, decide between separate indexes per tenant and a shared index with enforced filtering, and justify the choice based on expected tenant count.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Relying on application code to remember to filter by tenant ID on every query is not real multi-tenant isolation; a single missed filter in one code path can leak one tenant's data into another's search results, so isolation needs to be enforced at the infrastructure level, not just by convention.
