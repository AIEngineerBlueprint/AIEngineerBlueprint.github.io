Embeddings · Chapter 10

Evaluation

Evaluation of embeddings measures whether the vector space actually retrieves the right content for real queries, using metrics like recall at k and mean reciprocal rank against a labeled set of query-to-correct-answer pairs, rather than trusting an embedding model's general reputation.

Calculating reading timeHands-on labQuiz

In plain terms

It is like testing a librarian's filing system by asking real questions and checking whether the right book comes back, not just trusting that the filing system looks organized.

For non-technical readers

For a business reader, this is the difference between a search feature that 'seems to work' when a few people tried it, and one that's actually been proven to find the right answer reliably. Before trusting an AI search feature for anything customer-facing, ask whether it's been tested against a real set of questions with known correct answers, and what percentage it got right, rather than relying on someone's impression that it 'felt pretty good.'

How it works

In practice it comes down to building a labeled evaluation set of realistic queries paired with the correct document or chunk, running retrieval for each query, and measuring recall at k, meaning whether the correct answer appears in the top k results, alongside mean reciprocal rank for how highly it's ranked.

A concrete example

An evaluation set of 50 real customer questions paired with their correct support article measures that the current embedding model achieves 80 percent recall at 5, meaning the right article appears in the top 5 results 80 percent of the time.

Trade-offs

  • The core tension: A rigorous labeled evaluation set takes real effort to build and maintain, but without it, teams have no way to know whether switching embedding models or chunking strategies actually improved retrieval versus just changed it

Try it in code

const evalSet = [{ query: "how do I reset my password", correctChunkId: "faq-12" }, /* ... */];
let hits = 0;
for (const { query, correctChunkId } of evalSet) {
  const top5 = await retrieve(query, { topK: 5 });
  if (top5.some(r => r.id === correctChunkId)) hits++;
}
console.log("recall@5:", hits / evalSet.length);

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Build a 10-query evaluation set with the correct answer labeled for each, run it against your current retrieval setup, and calculate recall at 5.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • A model's general benchmark ranking on a public leaderboard does not guarantee it performs best on your specific domain and query patterns; only evaluation on your own realistic queries and documents tells you that.
