Distance metrics

Distance metrics are the mathematical functions, such as cosine similarity, Euclidean distance, and dot product, used to measure how close two embedding vectors are, and the right choice affects both retrieval quality and which vector index types are available.

Calculating reading timeHands-on labQuiz

In plain terms

It is like choosing between measuring how far apart two points are in a straight line, versus measuring the angle between them from a shared origin; both are valid distance concepts but answer different questions.

How it works

In practice it comes down to cosine similarity measuring the angle between vectors regardless of length, Euclidean distance measuring straight-line distance in the vector space, and dot product combining both angle and magnitude, each producing different rankings for the same set of vectors.

A concrete example

For text embeddings normalized to unit length, cosine similarity and Euclidean distance produce the same ranking, but for unnormalized embeddings from a model trained with a dot-product objective, using cosine similarity instead can quietly return worse rankings.

Trade-offs

  • The core tension: Cosine similarity is robust to magnitude differences and is the safest default for most text embeddings, but if an embedding model was specifically trained and evaluated using dot product, switching metrics without realizing it can degrade retrieval quality with no obvious error

Try it in code

const metrics = {
  cosine: (a, b) => dot(a, b) / (norm(a) * norm(b)),
  euclidean: (a, b) => Math.sqrt(a.reduce((s, v, i) => s + (v - b[i]) ** 2, 0)),
  dotProduct: (a, b) => dot(a, b)
};
// Use whichever metric your embedding model was trained/evaluated with.

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Hands-on lab

Task: Check which distance metric your embedding model's documentation recommends, and verify that your vector index or database is configured to use that same metric.

  1. Build or configure it, using the code example above as your starting point.
  2. Test it against one normal case and one edge case, and write down both actual outputs.
  3. Note what you would log, and which single metric would reveal this breaking in production.

Done when: the edge case is handled correctly or fails safely, not just the normal case.

Common mistakes

  • Distance metric choice is not a purely mathematical afterthought; it must match how the embedding model was actually trained and evaluated, or retrieval quality degrades silently with no error message to point to the cause.
