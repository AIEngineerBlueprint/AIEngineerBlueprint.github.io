Application Insights

Azure Application Insights, part of Azure Monitor, collects application telemetry such as request latency, dependency calls, and exceptions, giving AI systems visibility into model-call latency, error rates, and downstream dependency health. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

Beginner explanation

If you have no AI background, start here: Azure Application Insights, part of Azure Monitor, collects application telemetry such as request latency, dependency calls, and exceptions, giving AI systems visibility into model-call latency, error rates, and downstream dependency health.

Analogy: It is like a flight recorder for every request your application handles, showing exactly which internal calls it made and how long each took.

Core concept you must understand

Azure Application Insights, part of Azure Monitor, collects application telemetry such as request latency, dependency calls, and exceptions, giving AI systems visibility into model-call latency, error rates, and downstream dependency health.

What to learn

  • Core mechanism: an instrumentation SDK or auto-instrumentation capturing requests, dependencies such as calls to Azure OpenAI, and exceptions as structured telemetry, queryable and alertable through Azure Monitor.
  • Inputs and outputs: identify what enters Application Insights, what state changes, and what the next system component receives.
  • Evaluation signal: test Application Insights with representative success cases, edge cases, and at least one failure case.
  • Operational concern: assign ownership for freshness, permissions, cost, latency, and rollback where this concept touches production.
  • Debugging move: capture traces or artifacts that show whether Application Insights worked for the right reason.

Common misunderstandings

  • Having Application Insights enabled by default instrumentation is not the same as having real observability into an AI feature; custom telemetry for token usage, retrieval hit rate, or answer quality signals has to be added deliberately.
  • Application Insights is not complete when the happy path works once; it needs representative evals and visible failure handling.
  • Do not hide Application Insights inside prompt text when it requires code, policy, data, or operational ownership.

How to think about it

When to use it: Application Insights matters because it changes product quality, reliability, safety, cost, or developer velocity in real AI systems.

When it breaks: Having Application Insights enabled by default instrumentation is not the same as having real observability into an AI feature; custom telemetry for token usage, retrieval hit rate, or answer quality signals has to be added deliberately.

What to ask: What are the inputs, what evidence or data makes the result trustworthy, how will failure be detected, and what should the product do when confidence is low?

Trade-offs

  • Quality vs latency: The main trade-off is rich dependency tracing pinpoints exactly where latency or errors originate, but verbose instrumentation on every call adds real telemetry cost and volume that should be scoped to what's actually actionable
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.

Small runnable example

Use this snippet as the smallest concrete anchor for the concept. Change the inputs, then observe which assumptions fail.

// Track the Azure OpenAI call as a dependency
const start = Date.now();
const result = await openaiClient.getChatCompletions(deployment, messages);
telemetryClient.trackDependency({
  name: "azure-openai-chat", duration: Date.now() - start,
  success: true, dependencyTypeName: "HTTP"
});

Set up and test Application Insights in Azure

Make the first win concrete: prove identity, permissions, endpoint reachability, and one expected model or service response before building a full app.

Prerequisites

  • Confirm subscription, resource group, region, identity, role assignment, and networking before building.
  • Know whether the service uses keys, managed identity, private endpoints, or public endpoints.
  • Record resource names, deployment names, API versions, and Application Insights locations.

Smoke test

az account show --query "{subscription:id, tenant:tenantId}"
az group list --query "[0].name"
# Then run the smallest read-only command for the service in this chapter and save the request id.

How to know it worked

  • Authentication failures usually mean the wrong tenant, subscription, role, key, or managed identity.
  • Not found errors often mean wrong resource group, region, deployment name, or API version.
  • A useful smoke test records identity, resource, API version, latency, and expected output.

Next step: Only after identity and network access are proven should you add production prompts, data, or write actions.
