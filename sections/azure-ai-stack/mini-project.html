Azure AI Stack · Chapter 15

Mini project

The mini project ships one small, real AI feature deployed on Azure with documented managed identities, network configuration, and monitoring, that another engineer could redeploy from the instructions alone.

Calculating reading timeBuild project

What you're building, and why

In practice it comes down to packaging infrastructure as code or a clear manual runbook, the managed identities used, one Application Insights alert, and network configuration into a reusable, documented unit rather than a one-off portal click-through.

It is like handing off a fully labeled, working prototype instead of a working demo that only you know how to rebuild.

The brief

Take one small AI feature you've deployed on Azure, write it as infrastructure as code or a clear runbook, and have a teammate redeploy it using only your documentation.

A worked reference

A mini project might deploy a document-summarization API on Azure Functions with Blob Storage, Azure OpenAI inference, one Application Insights alert, and an infrastructure-as-code template a teammate can redeploy in a different subscription.

Set up and test Mini project in Azure

Make the first win concrete: prove identity, permissions, endpoint reachability, and one expected model or service response before building a full app.

Prerequisites

  • Confirm subscription, resource group, region, identity, role assignment, and networking before building.
  • Know whether the service uses keys, managed identity, private endpoints, or public endpoints.
  • Record resource names, deployment names, API versions, and Application Insights locations.

Smoke test

az account show --query "{subscription:id, tenant:tenantId}"
az group list --query "[0].name"
# Then run the smallest read-only command for the service in this chapter and save the request id.

How to know it worked

  • Authentication failures usually mean the wrong tenant, subscription, role, key, or managed identity.
  • Not found errors often mean wrong resource group, region, deployment name, or API version.
  • A useful smoke test records identity, resource, API version, latency, and expected output.

Next step: Only after identity and network access are proven should you add production prompts, data, or write actions.

Starting point in code

# main.bicep (excerpt) — redeployable by a teammate
resource ragFunction 'Microsoft.Web/sites@2023-01-01' = { }
resource ragStorage 'Microsoft.Storage/storageAccounts@2023-01-01' = { }
resource insightsAlert 'Microsoft.Insights/metricAlerts@2018-03-01' = { }
// az deployment group create --template-file main.bicep

Change the inputs and watch which assumption breaks first. The gap between what you expected and what you got is where the learning is.

Trade-offs to document

  • The core tension: Documenting and packaging the deployment takes longer than getting it working once in the portal, but an undocumented deployment is not reusable or auditable by anyone but its author

Definition of done

Watch out for

  • A feature working in your own Azure subscription is not the same as a shippable mini project; without infrastructure as code or a runbook another engineer can follow, the work is not transferable.
