Deployment patterns

Deployment patterns are the different ways a trained machine learning model is actually put into production use, such as batch predictions run periodically, real-time predictions served on demand, or embedded predictions running directly on a user's device. This chapter turns that idea into a traceable learning slice with examples, checks, and failure modes.

Learning objectives

  • Explain why Deployment patterns is important when building AI systems for users, not demos.
  • Describe the internal flow from input signals to model behavior, tool execution, or product outcome.
  • Choose architecture patterns and trade-offs for latency, accuracy, safety, cost, and maintainability.
  • Implement a small production-shaped slice with evaluation, observability, and failure handling.

For non-technical readers

The business-relevant question here is cost versus need for speed. Real-time AI feels more impressive, but it's meaningfully more expensive to run than a system that just recalculates predictions once a day or once a week. Before your team defaults to real-time, ask whether the actual business decision this feeds into truly needs instant answers, or whether yesterday's answer would work just as well at a fraction of the cost.

Trade-offs

  • Quality vs latency: The main trade-off is real-time serving gives the freshest, most interactive predictions but costs more per prediction and requires more serving infrastructure, while batch prediction is cheaper and simpler to operate but introduces staleness between runs
  • Flexibility vs reliability: Open-ended designs adapt to more inputs; constrained contracts are easier to test and operate.
  • Automation vs oversight: More automation reduces manual work but needs stronger logging, review points, and rollback paths.
